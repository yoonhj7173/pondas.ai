## 확정 (Confirmed)

### D1. MVP = 템플릿 기반 + 커스터마이즈 (2026-06-11)
- **결정:** 미리 만든 팀 템플릿(기획/리서치/디자인/개발/데이터)을 제공. 유저는 그 안에서
  **팀 추가/삭제, 에이전트 추가/삭제, 역할(= system prompt/instruction) 수정** 가능. 백지 생성 아님.
- **근거:** solo founder의 첫 경험에서 백지는 막막. 템플릿 → SCV식 변형이 자연스럽다.
- **백엔드 영향:** 현재 `clusters`/`units`는 고정 seed 픽스처 → **유저 소유 + 편집 가능 row**로 전환.
  에이전트별 편집 가능 instruction 필드 추가. item 1~4 수정 필요.

### D3. 오케스트레이터 = 프로젝트당 별도 챗, cross-crew P0 (2026-06-11)
- **결정:** 커맨드센터/오케스트레이터 에이전트는 맵 위 유닛이 아니라 **별도 챗 UI**.
  **프로젝트당 1개.** 모든 팀에 일을 뿌리고 유저와 freeform 대화. cross-crew 라우팅을 **P1 → P0**로 승격.
- **백엔드 영향:** 오케스트레이터는 project 스코프. cross-crew dispatch가 P0 기능.

### D2. project ↔ team/agent 계층 = 프로젝트 격리 + 템플릿/인스턴스 분리 (2026-06-11)
- **결정:**
  - **템플릿** (시스템 라이브러리, read-only): 기획/리서치/디자인/개발/데이터 팀 + 에이전트 청사진. D1의 "미리 만든 팀".
  - **인스턴스** (프로젝트 종속, 편집 가능): 프로젝트 생성 시 템플릿을 **복제**해 프로젝트 내부에 실제 team/agent row 생성.
  - 팀/에이전트는 **프로젝트 격리** (공유 아님). context/memory/토큰사용량/로그도 전부 프로젝트 스코프.
- **근거:** 에이전트 가치는 누적된 프로젝트 컨텍스트에서 나옴 → 공유하면 결국 프로젝트별 분기 = Y로 수렴.
  StarCraft 메타포(프로젝트 = 맵 1개)와도 1:1.
- **백엔드 영향:** 모든 핵심 테이블에 `project_id` 스코프. 템플릿 테이블 신규. seed 4팀8유닛은 템플릿으로 재배치.

### D4. Output = 로컬 파일 (MVP) + 최소 목록/다운로드 어피던스 (2026-06-11)

### D14. Context/Memory = 2층 구조, RAG 없음 (MVP) (2026-06-11)
- **① 프로젝트 컨텍스트 (유저 업로드):** 시작화면 + settings에서 업로드. object storage(원본) + 텍스트 추출(DB).
  task 실행 시 추출 텍스트를 토큰 버짓 내 **풀텍스트 프롬프트 주입**. 벡터DB/RAG는 P1.
- **② 에이전트 메모리 (자동 누적):** 에이전트별 markdown 스크래치패드. task 종료 시 배운 것/결정사항 append,
  다음 task에 자동 주입. 유저는 settings에서 열람/편집/삭제 (= User 6 "메모리 관리 보조").
- **인프라 영향:** object storage 1개 추가. 나머지는 기존 Postgres로 충분.

### D16. Stop/Cancel + 프로젝트 일시정지 = P0 승격 (2026-06-11)
- **결정:** 에이전트별 **Stop**(실행 중 task 취소) + **프로젝트 전체 일시정지**(비상버튼)는 P0.
- **근거:** D6/D21로 그래프가 자동으로 일을 연쇄 발화시키고 loop까지 돎. 자동화 수준을 올렸으면
  브레이크도 같이 올라가야 함. 잘못 와이어링했을 때 유저의 유일한 방어가 cost cap이면 안 됨.

### D17. 에이전트 = 한 번에 task 1개 (2026-06-11)
- **결정:** 에이전트는 단일 작업. 동시에 task 2개를 돌지 않음. 병렬성이 필요하면 **에이전트를 더 뽑는다** (SCV 방식).
- **근거:** 오피스 메타포와 일치 (캐릭터 1명 = 일 1개), 애니메이션 = 상태 1개로 단순,
  그래프 실행 시 디스패치 규칙 명확 (대상 에이전트 busy면 큐잉).

### D18. 결과물 인라인 프리뷰 (D4 보완) (2026-06-11)
- **결정:** 결과물 목록에서 텍스트/마크다운 파일은 **클릭 시 읽기 전용으로 인라인 렌더링**. 다운로드는 유지.
- **경계:** "관리"는 여전히 없음 — 편집/버전/폴더정리/코멘트/검색 전부 ❌. 추가되는 건 읽기 전용 뷰어 하나.
- **근거:** 읽기와 관리는 다른 문제. 결과 하나 읽는 데 다운로드 4단계는 제품의 마지막 1미터가 끊기는 것.

### D19. loop 스코프 = 엣지 2종으로 제한 (D6 보완) (2026-06-11)
- **결정:** MVP 엣지 종류는 딱 둘.
  - **① handoff** — 단방향 파이프. A 완료 → A의 output이 B의 input으로, B 시작.
  - **② review loop** — 왕복. **최대 N회 유저가 숫자로 고정** + 리뷰어가 승인 시그널(예: `APPROVED` —
    검증된 sentinel 패턴 재활용) 내면 조기 종료. N회 내 승인 실패 시 loop 종료 + 리포트 → 유저 판단.
- **조건식 loop("~할 때까지 반복", LLM이 종료 판단)는 P1.** LLM 판단 종료는 1회 종료 아니면 무한 회전으로
  수렴하는 flaky의 온상. 고정 N회면 최악 비용도 예측 가능 (N × 2 × task당 비용).

### D20. Board 재정의 = 작업 계획 체크리스트 (D7 대체) (2026-06-11)
- **결정:** board = **implementation-plan.md 같은 작업 체크리스트 뷰.** 지시를 주면 오케스트레이터가
  작업 아이템들로 쪼개고, board는 지시(목표) 단위로 묶어 **어떤 아이템이 있고 / 뭐가 완료됐고 / 뭐가
  진행 중인지** 전체 그림을 보여줌. 그 이상도 이하도 아님.
- **설계 귀결:** 별도 "action item 프로토콜" 불필요 — **tasks 테이블에서 그대로 파생.** 이미 만들 데이터라
  추가 규약 zero, P0 유지. (D7의 "엔티티 설계 TBD"는 이걸로 해소.)

### D21. 라우팅 권한: 엣지 = 자동 파이프, 챗 = 진입점 + 일회성 오버라이드 (2026-06-11) ⭐
- **결정:**
  - **엣지는 항상 자동 발화.** A→B 엣지가 있으면 A가 task를 끝낼 때마다 무조건 B로 흐름.
    유저가 그린 그래프 = 실제 동작. (가끔만 작동하면 그림이지 메커니즘이 아님.)
  - **오케스트레이터 = 진입점 + 일회성 오버라이드.** 일은 챗으로 시작. 챗으로 다른 라우팅을 지시하면
    **그 task 1건에 한해** 챗이 우선하되 **엣지 자체는 불변** — 다음 완료부터는 다시 엣지대로.
- **근거:** 오케스트레이터가 매번 라우팅을 판단하면 같은 작업이 어제는 기획, 오늘은 디자인으로 가는
  비결정성 발생. **반복되는 구조 = 엣지(결정적), 예외/즉흥 = 챗(유연).** 분업.
- **P1 씨앗:** 같은 오버라이드 반복 시 오케스트레이터가 "이거 엣지로 만들까요?" 제안.

### D22. 챗으로 blocked 에이전트 응답 가능 (D10 보완) (2026-06-11)
- **결정:** 오케스트레이터 챗을 통해서도 blocked/needs-input 에이전트에 입력 전달(resume) 가능.
  에이전트 패널의 "Provide human input"과 병행 — 경로가 둘.
- **근거:** 유저는 분명 챗에 "그 QA가 물어본 거 B안으로 해"라고 침. 챗이 "패널 가서 하세요"라고
  답하면 멍청해 보임. 챗 = 만능 인터페이스라는 D3 철학과 일치.

### D23. 맵 위 상태 인디케이터: "!" (2026-06-11)
- **결정:** needs-input/blocked 시 **에이전트 머리 위에 "!" 아이콘** — 해결될 때까지 유지, 줌아웃에서도
  인지 가능 (펄스 등). `failed`도 동일 계열의 아이콘 (예: 빨간 표시)으로 일관성 유지.
- **근거:** 기존 알림 경로(토스트=순간, 이벤트 피드=우측)에 **맵 자체에 남는 시각 신호**가 없었음.
  토스트는 놓치면 끝, !는 해결까지 떠 있음. TPH/심즈 문법 그대로라 학습 비용 zero.

### D24. 퍼블릭 랜딩 페이지 + SEO/GEO = P0 (2026-06-11)
- **결정:** 앱과 **완전히 별개의 퍼블릭 랜딩 페이지**(`/`) 신설 — 단순 프로덕트 소개 (뭔지/누구를 위한 건지/
  스크린샷·데모/CTA→로그인). 흐름: 랜딩 → Clerk 로그인 → 게임식 시작화면(Flow 0) → 맵.
- **렌더링 분리:** 랜딩 = Next.js **SSG**(정적 생성, 크롤러 친화) / 앱 본체 = CSR (Pixi 캔버스, auth 뒤 —
  SEO 불필요·불가). 랜딩 번들에 Pixi가 딸려가지 않게 코드 스플리팅 (Core Web Vitals).
- **SEO 패키지 (MVP):** title/description, OG 태그+이미지, canonical, `sitemap.xml`, `robots.txt`,
  시맨틱 HTML, JSON-LD(SoftwareApplication).
- **GEO 패키지 (MVP):** `llms.txt`, AI 크롤러(GPTBot/ClaudeBot/PerplexityBot) robots 허용,
  인용하기 쉬운 명확한 평문 콘텐츠 구조.
- **블로그:** 인프라는 **D33으로 P0 승격** (어드민 전용 MDX 블로그). 고급 기능(태그/RSS/검색)만 P1.

### D25. handoff 엣지 = DAG 강제 (저장 시 사이클 거부) (2026-06-12)
- **결정:** 엣지 저장 시점에 handoff-only 서브그래프에 사이클이 생기면 **거부** + "review loop를 쓰세요" 안내.
  순환은 종료 조건이 내장된 review loop(D19)로만 표현 가능.
- **근거:** handoff에는 멈출 조건이 없어 원형이 되면 무한 회전 — chain budget/cost cap은 돈을 태운 뒤의
  사후 브레이크. 그리는 순간에 막는 게 맞다.
- **트레이드오프:** 의도적 다자 순환(A→B→C→A 다듬기)은 MVP에서 불가 — 쌍 review loop 또는
  오케스트레이터 수동 재지시로 우회. "다자 순환 + N 제한" 일반화는 P1 후보.

### D26. 오케스트레이터 = LiteLLM 기반 tool-loop (CrewAI 미사용) (2026-06-12)
- **결정:** 오케스트레이터는 CrewAI crew가 아닌 **직접 구현 tool-loop**, 단 호출 레이어는 **LiteLLM**
  (CrewAI의 기존 의존성 — 신규 패키지 없음). 모델은 config로 Claude 고정 (MVP).
- **근거:** CrewAI는 fire-and-forget task용이지 대화형 챗용이 아님 (프롬프트 비계 오염, 레이턴시,
  감사가능성). 모델 종속 우려는 LiteLLM의 provider 중립 포맷으로 해소 — 워커(CrewAI→LiteLLM)와
  오케스트레이터가 같은 호출 경로라, **추후 유저 모델 설정 기능 = 리팩토링이 아니라 config/UI 작업.**
- **잔여 debt:** 모델별 가격표(config 구조)만. prompt caching은 LiteLLM passthrough로 사용.

### D27. 파일 저장 = Postgres + FileStore 인터페이스 (MVP) (2026-06-12)
- **결정:** outputs(코드 파일트리·zip 포함)와 context 원본은 **Postgres**(text/bytea)에 저장,
  `FileStore` 인터페이스 뒤에 격리. object storage(R2 등)는 MVP에서 **사용 안 함** — 인프라 0개 추가.
- **근거:** MVP 산출물은 텍스트/코드 위주 수백 KB 규모라 Postgres로 충분. 라이브 작업 파일은
  샌드박스 워크스페이스가 보유(D29 pause로 보존). P1(실배포·바이너리 아티팩트 시대)에 `S3FileStore`로
  교체 — 인터페이스 덕에 국소 작업.
- ※ 유저 명시 확정이 아닌 어시스턴트 추천 채택 (A7′ 논의의 귀결). 이의 시 변경 용이.

### D28. MVP = 옵션 B (실행 포함) + 투트랙 빌드 (2026-06-12)
- **결정:** MVP는 실행 능력 포함 — "실행 없는 MVP-A는 검증할 가치가 없다" (개발팀이 가짜면 타겟인
  솔로파운더에게 줄 가치가 없음). 페이즈 분리는 가설검증 관점이 아니라 **개발 복잡성 관리 관점**으로:
  - **Track 1 — 앱**: 기존 플랜 (스키마/그래프/오케스트레이터/맵/패널/보드/랜딩)

    완전 격리, UI 없이 단독 개발·테스트 가능. 마지막에 통합.
- **정합성:** cc-v1 워크플로우가 기존 설계에 1:1 매핑 — 기능분해=goal/board(D20), 한 기능씩=D17,
  qa↔dbg 루프=review loop(D19), 단계 핸드오프=handoff(D21). 빠진 건 실행뿐이었음.

### D29. 샌드박스 = E2B (어댑터 뒤) (2026-06-12)

### D47. 타겟 재정의 — "Claude Code-aware builder" (2026-07-07)
- **배경 (런칭 후 제품 재평가):** 기존 암묵 타겟 "멀티에이전트/하네스를 쓰고 싶은데 터미널이 어려운

  만들기** (문서/리서치는 그 안에 포함). 솔로파운더 페르소나는 이 코호트의 부분집합으로 유지.

  **유저가 실제 소유한 것**(repo/문서/채널) 위에서 일할 때 생김. ① 이 코호트에서 utility 증명 →
  ② 유저 실컨텍스트 연동(GitHub, 이후 MCP) → ③ 일반인 확장. ②를 건너뛴 일반인 GTM은 "귀엽네 → 3일 뒤
  이탈".
- **경쟁 프레임:** 이 타겟에선 경쟁이 Claude Code 생태계 자체(병렬 세션, Conductor류). 플랫폼 리스크는

### D48. Phase 2 = closure-first — "받는 경험"이 MVP 완성 조건 (2026-07-07)
- **진단 (코드 레벨 확인):** 제품은 "일이 진행되는 걸 구경하는 경험"만 팔고 "완성된 결과물을 받는

  렌더도 없음) ③ AgentPanel은 task 완료 후 결과 자체를 안 보여줌 ④ 가입 240크레딧 < strong task

  **오피스 캔버스는 손대지 않음** (바이럴 자산 완성 상태).

  `npm run dev` → public URL 노출. 엔진이 뭐든(cma/e2b) 프리뷰 동일 동작, D45 불변.
- **수명 정책 (파운더 확정, 비용=정책):** on-demand 기동(시어터 열 때/dev task 완료 시) → **idle 10분

  최대 1개. 샌드박스 가동 분은 task/프로젝트에 로깅(원가 가시성).
